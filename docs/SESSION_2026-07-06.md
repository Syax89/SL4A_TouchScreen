# Debug Session 2026-07-06 — Final Summary

---

## 1. Goal

Understand why SPI writes don't work on Linux (Surface Laptop 4 AMD, device MSHW0231).

**Final verdict**: Software EXHAUSTED. Logic analyzer needed.

---

## 2. Tools used

- **parse_spi.py** — full analysis of the ETW CSVs captured from Windows
- **gpio_test.ko** — kernel module to test M009/M010 from the command line
- **RWEverything** (Windows) — MMIO dump of the AMD SPI controller
- **mmio_write.c** — raw MMIO kernel module: Windows-exact write + FIFO dump

---

## 3. Tests performed (in order)

### 3.1 Windows CSV analysis
Comparison of `surface_init.csv` vs `surface_boot_auto.csv` vs `surface_touch.csv`:
- Cold boot: **vendor init** to register 0x04 (14 bytes), device goes straight to DATA, NO DESCREQ
- Warm boot: RESET_RSP → DESCREQ → DEVICE_DESC → RPT_DESC → DATA
- Touch runtime: 4304B full frames, 216B heatmap frames, alternating every ~10 frames
- Device descriptor: Vendor=0x045E, Product=0x0C19, Version=0x0100
- Approval bytes are an ECHO of the last write, not protocol constants
- Companion device on a separate SPB connection (0x18/0x19/0x1a): the firmware upload is NOT for the touchscreen

### 3.2 Vendor init implemented
- Added the vendor init `02 00 00 04 82 00 00 04 00 01 01 0C EE 5B` to the probe of `spi-hid-core.c`
- **Result**: MISO changed from `[00 00 00...]` to `[ff 00 ff 00... 03 00 00]` — the device responds partially
- But DESCREQ is still KO

### 3.3 Full-duplex with 2 separate transfers
- Write transaction (TX only) followed by a separate read transaction
- **Result**: stable, MISO changed, no crash
- Adopted as the current mode

### 3.4 _PS0 / _INI / _RST via ACPI
- Explicit calls to the ACPI functions for device power-on and reset
- **Result**: no effect on writes

### 3.5 M009 / M010 GPIO test
- gpio_test.ko: GPIO read/write via ACPI evaluate
- M009(0x5B)=1 → power rail active
- M009(0x0103)=1 → reset de-asserted
- HW power cycle (M010 OFF/ON): **no effect on writes**

### 3.6 WINSEQ test
- Exact Windows sequence (per the hidspi.sys decomp) in the probe
- **Read path**: RESET_RSP header+body identical to Windows → **OK**
- **Write path**: DESCREQ ignored → **KO**

### 3.7 Windows MMIO dump via RWEverything
- 4 dumps: IDLE (no touch), TOUCH1, TOUCH2, TOUCH3
- **Critical discovery**: CTRL0 byte[15:8] = 0x0E on Windows, 0xA9 on Linux

### 3.8 CTRL0 fix
- Attempted to write 0x0E into byte[15:8] of CTRL0
- **Result**: the controller **ignores** every write to that byte — an immediate read after the write still shows 0xA9
- The value is **hardwired** by the BIOS, not modifiable from software

### 3.9 CTRL1 fix
- Attempted to write 0x020006B5 (the Windows value)
- **Result**: CTRL1 is **read-only**, every write is ignored

### 3.10 Full FIFO dumps (mmio_write.c)
- Write: correct TX data in the FIFO, MISO = ALL ZEROS (70 bytes) — the device doesn't drive the line
- Read: MISO contains valid HID data
- The device **selectively ignores** writes

### 3.11 ACPI verification
- DSDT md5sum identical between Windows/Linux: `78046fa74c0282ee59db8b04a5204d88`
- ACPI tables bit-identical — no hardware configuration difference

## 5. Final Conclusions

- **Read path**: working, bit-identical to Windows
- **Write path**: EXHAUSTED. The AMD FCH Cezanne controller under Linux does not produce a valid write signal recognized by the MSHW0231 touchscreen
- **CTRL0 bits[15:8] hardwired to 0xA9** (0x0E on Windows) is the prime suspect
- **CTRL1 is read-only** — the Windows value can't be reproduced
- **No software fix is possible** without understanding the physical signal difference
- **MANDATORY next step**: logic analyzer on SCK/MOSI/MISO/CS between Windows and Linux
