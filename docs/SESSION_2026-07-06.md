# Sessione Debug 2026-07-06 — Riepilogo Finale

---

## 1. Obiettivo

Capire perche' le write SPI non funzionano su Linux (Surface Laptop 4 AMD, device MSHW0231).

**Verdetto finale**: Software ESAURITO. Serve logic analyzer.

---

## 2. Tool utilizzati

- **parse_spi.py** — analisi completa dei CSV ETW catturati da Windows
- **gpio_test.ko** — modulo kernel per testare M009/M010 da riga di comando
- **RWEverything** (Windows) — dump MMIO del controller AMD SPI
- **mmio_write.c** — modulo kernel raw MMIO: write Windows-exact + FIFO dump

---

## 3. Test eseguiti (in ordine)

### 3.1 Analisi CSV Windows
Confronto `surface_init.csv` vs `surface_boot_auto.csv` vs `surface_touch.csv`:
- Cold boot: **vendor init** a registro 0x04 (14 byte), device entra in DATA immediato, NO DESCREQ
- Warm boot: RESET_RSP → DESCREQ → DEVICE_DESC → RPT_DESC → DATA
- Touch runtime: 4304B full frames, 216B heatmap frames, alternanza ogni ~10 frame
- Device descriptor: Vendor=0x045E, Product=0x0C19, Version=0x0100
- Approval bytes sono ECO dell'ultima write, non costanti di protocollo
- Companion device su connessione SPB separata (0x18/0x19/0x1a): firmware upload NON e' per il touchscreen

### 3.2 Vendor init implementato
- Inserito nel probe di `spi-hid-core.c` il vendor init `02 00 00 04 82 00 00 04 00 01 01 0C EE 5B`
- **Risultato**: MISO cambiata da `[00 00 00...]` a `[ff 00 ff 00... 03 00 00]` — il device risponde parzialmente
- Ma DESCREQ ancora KO

### 3.3 Full-duplex con 2 transfer separati
- Write transaction (TX only) seguita da read transaction separata
- **Risultato**: stabile, MISO cambiata, nessun crash
- Adottato come modalita' corrente

### 3.4 _PS0 / _INI / _RST via ACPI
- Chiamate esplicite alle funzioni ACPI per power-on e reset del device
- **Risultato**: nessun effetto sulle write

### 3.5 M009 / M010 GPIO test
- gpio_test.ko: lettura e scrittura GPIO via ACPI evaluate
- M009(0x5B)=1 → power rail attivo
- M009(0x0103)=1 → reset de-asserted
- Power cycle HW (M010 OFF/ON): **nessun effetto sulle write**

### 3.6 WINSEQ test
- Sequenza esatta Windows (come da decomp hidspi.sys) nel probe
- **Read path**: RESET_RSP header+body identici a Windows → **OK**
- **Write path**: DESCREQ ignorato → **KO**

### 3.7 Dump MMIO Windows via RWEverything
- 4 dump: IDLE (nessun touch), TOUCH1, TOUCH2, TOUCH3
- **Scoperta critica**: CTRL0 byte[15:8] = 0x0E su Windows, 0xA9 su Linux

### 3.8 CTRL0 fix
- Tentativo di scrivere 0x0E nel byte[15:8] di CTRL0
- **Risultato**: il controller **ignora** ogni write a quel byte — read immediato dopo la write mostra ancora 0xA9
- Il valore e' **hardwired** dal BIOS, non modificabile da software

### 3.9 CTRL1 fix
- Tentativo di scrivere 0x020006B5 (valore Windows)
- **Risultato**: CTRL1 e' **read-only**, ogni write ignorata

### 3.10 Dump FIFO completi (mmio_write.c)
- Write: dati TX corretti in FIFO, MISO = TUTTI ZERI (70 byte) — device non pilota la linea
- Read: MISO contiene dati HID validi
- Il device **ignora selettivamente** le write

### 3.11 Verifica ACPI
- DSDT md5sum identico Windows/Linux: `78046fa74c0282ee59db8b04a5204d88`
- Tabelle ACPI bit-identiche — nessuna differenza di configurazione hardware

---

## 5. Conclusioni Finali

- **Read path**: funzionante e bit-identical a Windows
- **Write path**: ESAURITO. Il controller AMD FCH Cezanne sotto Linux non produce un segnale write valido che il touchscreen MSHW0231 riconosce
- **CTRL0 bits[15:8] hardwired a 0xA9** (0x0E su Windows) e' il principale sospettato
- **CTRL1 e' read-only** — il valore Windows non e' riproducibile
- **Nessun fix software e' possibile** senza capire la differenza di segnale fisico
- **Next step OBBLIGATORIO**: logic analyzer su SCK/MOSI/MISO/CS tra Windows e Linux
